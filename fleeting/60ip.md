# Firefox camera preview uses direct V4L Lenovo MJPG

Timestamp: 2026-05-08T00:10:35-07:00

Question / hypothesis: While Firefox camera preview is actively streaming, determine whether it uses PipeWire video or direct V4L, and identify the exact negotiated camera mode that feels low latency.

Method: With Firefox preview active, inspected PipeWire via wpctl status, pw-cli ls Node, and pw-cli ls Device. Checked V4L users with fuser on /dev/video49-56 and lsof /dev/video53. Queried active V4L state with v4l2-ctl -d /dev/video53 --all.

Observed result: PipeWire still showed no Video devices, Video sources, or Video streams. Firefox appeared only as PipeWire audio input/output streams. fuser showed Firefox PID 2785 using /dev/video53 directly. lsof confirmed Firefox has /dev/video53 open. v4l2-ctl showed /dev/video53 active as Lenovo Performance Camera, UVC driver, 1280x720 MJPG, 30 fps. Auto exposure remained Aperture Priority Mode, exposure_time_absolute inactive.

Interpretation: Firefox's low-latency preview is not coming from a visible PipeWire video path on this system. It is using direct V4L on /dev/video53 with 1280x720 MJPG at 30 fps. We should mirror Firefox's direct V4L configuration and focus on buffer policy, queue timing, frame dropping, and render/display phase rather than adding PipeWire/pw-v4l2 to the latency-critical path.

Follow-up tasks / instrumentation gaps: Make tron-stream defaults match Firefox's observed mode: Lenovo RGB /dev/video53, 1280x720 MJPG 30 fps. Compare direct V4L at that mode with buffers 1/2/4 and drain-latest by eye. If possible later, use strace/ioctl tracing on Firefox to see requested V4L buffer count and streaming setup.
