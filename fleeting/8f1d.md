# Drain-to-latest did not find V4L backlog in steady MJPG test

Timestamp: 2026-05-07T23:56:44-07:00

Question / hypothesis: If multi-buffer V4L latency comes from queued stale frames, a drain-to-latest loop should observe already-ready buffers and skip stale candidates before publishing.

Method: Added --drain-latest to tron-stream. In drain mode, capture blocks for one frame, copies that candidate payload, polls the V4L handle with timeout 0 for another ready buffer, and if one exists skips decode/publish for the stale candidate and repeats. Tested Lenovo RGB /dev/video53 at 640x480 MJPG 30 fps with TurboJPEG, --buffers 4 --drain-latest.

Observed result: The test stayed near 30 fps with V4L timestamp deltas around 33.3 ms and no sequence gaps. The drained counter stayed at 0 in the steady run, while render-side age varied by render/camera phase. This means no already-ready backlog was visible to userspace in this particular steady-state path.

Interpretation: Drain-to-latest is still useful as a latency policy if processing ever falls behind, but it did not explain the perceived buffer-count difference in the simple 640x480 MJPG test. The visible single-buffer improvement may come from queue-depth effects before userspace-visible readiness, UVC/driver scheduling, camera internal buffering, or subjective/manual test bias. We need a physical photon-to-display measurement before treating the perceived difference as proven.

Follow-up tasks / instrumentation gaps: Keep --drain-latest available for HID experiments and compare subjectively with --buffers 1 and --buffers 4. Add a true latency measurement rig; internal V4L sequence/timestamp counters are not enough to validate photon-to-frame latency.
