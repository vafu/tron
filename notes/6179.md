# Proper stereo calibration direction

Timestamp: 2026-05-07T20:20:41-07:00

Tags: #tron #task

Question / hypothesis: Why did the first RGB/IR checkerboard calibration work
poorly, and what should a proper stereo calibration flow do instead?

Observed result: OpenCV's stereo path is valid for RGB/IR, but the capture
protocol must be much stricter. The current flow accepts any pair where both
views detect the board, with no coverage scoring, no sample diversity check, no
per-view reprojection filtering, and only three samples required. OpenCV docs
and stereo calibration tools emphasize synchronized pairs, many diverse poses,
wide X/Y coverage, size/depth variation, tilted/skewed board poses, subpixel
corner refinement, independent intrinsics, then stereo extrinsics. OpenCV Rust
bindings expose `calibrate_camera_extended`, `stereo_calibrate_extended`, and
`find_chessboard_corners_sb`, so per-view errors and a stronger detector are
available locally.

Follow-up tasks / instrumentation gaps: Rework `tron-calib` around calibration
quality gates: require many samples, show X/Y/size/skew coverage, reject similar
samples, compute per-view errors, remove bad views, and only then save a stereo
profile. Consider ChArUco/asymmetric circles if plain chessboard remains fragile
in IR; this may need OpenCV `aruco`/contrib support.
