# Tron pointer research kickoff

Timestamp: 2026-05-02T13:13:21-07:00

Tags: #tron

Working title: Leverage Windows Hello camera stack to use as a 3D/2D pointer system.

Question:

Can the existing Tron RGB/IR/proximity pipeline evolve into a practical 3D/2D pointer system, starting with a demo that tracks `x`, `y`, and `z` well enough to drag a cube in 3D space?

Near-term demo:

- Keep the existing camera/debug views visible.
- Add a 3D cube.
- Add a hand or pointer representation in the same 3D scene.
- Estimate three axes from the current hand pipeline plus IR/proximity cues.
- Allow the hand representation to grab and drag the cube.

Research direction:

The demo is not the final goal. It is an instrumented stepping stone toward understanding what data, labels, and model architecture are needed for a custom pointer model. The eventual target is reliable 2D and 3D pointing using a Windows Hello-style camera stack.

Initial architecture notes:

- Camera capture must remain backend-swappable. V4L is current; libcamera/IPU6 should not require a broad refactor.
- ONNX Runtime must remain centralized so GPU inference/provider work can be added in one place.
- Tracking, ROI acquisition, landmark extraction, smoothing, and gesture/pointer interpretation should remain independently replaceable stages.
- Rendering should consume a clean pointer/hand state rather than depending on model internals.

Follow-ups:

- Define `PointerState` with normalized `x`, `y`, `z`, confidence, and interaction state.
- Decide how to infer initial `z`: landmark scale, IR foreground size/intensity, proximity sensor, or a fused heuristic.
- Add a 3D scene path in `gfx` without breaking the current camera debug panes.
- Add latency/jitter instrumentation before evaluating depth quality.
- Plan data capture and labeling once the cube demo can produce repeatable interactions.
