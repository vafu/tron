# Raw camera stream avoids RGBA conversion

Timestamp: 2026-05-07T22:20:15-07:00

Question / hypothesis: Is the stream lag caused by converting YUYV/GREY frames into cloned RGBA images before render?

Observed result: A dedicated tron-stream debug binary now uploads native camera bytes directly to wgpu textures and converts only in shader: YUYV uses Rg8Unorm with width*2 bytes per row and GREY uses R8Unorm. The capture thread keeps a single reusable RawFrame buffer behind Arc<Mutex<_>> and publishes by extending that buffer from the V4L mmap slice. The render path borrows the RawFrame only for queue.write_texture, then releases the lock before surface acquire/render/present. The same GPU texture can be drawn repeatedly by binding it to any number of panes; there is no need to clone image data per pane.

Smoke test: Lenovo RGB /dev/video53 negotiated 1024x576 YUYV. GPU upload was about 0.02-0.04 ms, render about 0.08-0.10 ms, present about 16.5 ms due to vsync. Camera delivery was only about 10-12.5 fps. Lenovo IR /dev/video55 negotiated 640x480 GREY. GPU upload was about 0.014 ms, render about 0.07-0.12 ms, present about 16.5 ms, camera delivery 15 fps.

Interpretation: In the raw path, GPU upload/render are cheap. Remaining visible latency is dominated by selected camera frame rate and vsynced presentation, not CPU YUYV-to-RGBA conversion. Main pipeline still has the old camera backend that decodes to RGBA and clones every frame; that should be replaced by a raw frame abstraction rather than optimized locally.

Follow-up tasks: Promote RawFrame/native format into camera module; make render panes borrow the same frame/texture; add framerate-aware mode selection instead of picking only by resolution/aspect; consider Mailbox or Immediate present mode for latency tests; keep CPU RGBA conversion only for consumers that explicitly need CPU RGB bytes.
