# HTTP diagnostics metadata publisher

Timestamp: 2026-05-07T18:13:59-07:00

Tags: #tron #task

Question / hypothesis: Publishing per-frame pipeline metadata over a small HTTP
server should make live debugging less noisy than terminal logs and keep the
diagnostics UI separate from `main.rs`.

Observed result: Added a `diagnostics` module with a local HTTP server and a
snapshot state handle. The gesture pipeline now publishes frame seqs, timing,
proximity, hand ROI/classifier state, IR depth cue metrics, and pointer state to
`/state.json`; `/` renders a polling browser UI. `cargo check --bin tron`
passes.

Follow-up tasks / gaps: Validate the live server while cameras are streaming and
decide which additional metadata belongs in the snapshot once the depth cue
experiment stabilizes.

Update: 2026-05-07T18:18:14-07:00

Question / hypothesis: Port `8765` conflicts with an existing local service, so
the diagnostics default should avoid it while still supporting explicit port
selection.

Observed result: Moved the default diagnostics port to `8766`. The default path
now falls forward through a small local port range if the preferred port is
occupied; explicit `--diagnostics-port PORT` remains exact.

Follow-up tasks / gaps: If multiple tron instances are run simultaneously, the
browser URL should be read from the startup log because the fallback port may
vary.
