# IR exposure control research

Timestamp: 2026-05-02T17:43:48-07:00

Tags: #tron

Question / hypothesis:

Can the IR camera exposure be controlled or at least observed on Linux when
`/dev/video2` does not expose simple V4L exposure sliders?

Observed result:

Initial research suggests a three-layer control model:

- Standard V4L2 camera controls are the easiest path when present:
  `exposure_auto`, `exposure_absolute`, `exposure_auto_priority`, `gain`,
  `3a_lock`, exposure metering, and possibly UVC ROI auto-exposure controls.
- Some Windows Hello-style IR cameras expose emitter or sensor controls only as
  UVC extension-unit controls. These will not necessarily appear as ordinary
  `v4l2-ctl --list-ctrls` sliders unless mapped into V4L2 controls.
- The local `linux-enable-ir-emitter` submodule already probes UVC XU controls
  directly with `UVCIOC_CTRL_QUERY`, scanning unit/selector pairs, reading
  current/min/max/default values, and testing writability. That mechanism is
  relevant for hidden emitter brightness or exposure-adjacent controls.

The live `/dev/video2` enumeration was not captured yet because device probing
was not approved in this session.

Follow-up tasks / instrumentation gaps:

- Run `v4l2-ctl -d /dev/video2 --all` and
  `v4l2-ctl -d /dev/video2 --list-ctrls-menus`.
- Run `linux-enable-ir-emitter --grey-devices` to dump UVC XU controls for
  grayscale devices.
- If standard exposure controls are absent, build a small read-only XU dump in
  Tron by reusing the submodule's unit/selector query approach.
- Add IR telemetry: mean/percentile brightness, saturated pixel fraction,
  foreground mask coverage, flashlight-on state, and frame cadence.
- Treat unknown writable XU mutation as risky; prefer read-only enumeration
  before trying value changes.
